## 一、离现阶段（数据处理、索引构建、评估体系）

### 1.1 文档处理 & 切片策略

**Q1.** 500个文档，你用了什么切片策略？固定长度、固定token数、还是基于语义的切片？为什么选这个方案？你对比过哪几种切片方式的召回效果？
 
我们的文档主要是 Markdown 格式，面向内部用户的 API 文档、故障手册、功能说明和版本更新记录。基于这些文档结构化的特点，我们的切分策略是：先用
  Markdown parser 按一二级标题做结构拆分，然后在每个二级标题域内，如果内容超过设定的 token 上限，再递归地按段落和行进行切分。

  对于特殊文本，比如表格、JSON 配置、代码块，我们不会按上限硬截断，而是做两条处理：第一，在清洗阶段用 LLM
  离线为这些特殊块生成一段自然语言描述，拼入 chunk 的 content 中，补充语义信息；第二，对于非特殊文本的普通段落，我们会把当前 chunk
  所属的一二级标题路径前置拼接到 content 头部，增强 chunk 的上下文信息。这样即使 chunk 被单独召回，大模型也能知道这段话出自哪个文档的哪个章节。 

  关于 chunk size 的选择，我们对比了 256、384、512 三个档位。选型逻辑分两步。第一步，全量 500
  个文档做语义切分后统计截断率。截断的正确定义是：如果当前 chunk 长度触达上限，且它和相邻 chunk
  共享同一标题路径，说明它是被硬截断的，而不是刚好自然结束。我们 384 和 512 的截断率都在百分之一左右，差距不大。第二步，我们构造了 100 多条 QA  
  pairs 做检索召回评测。评测集是按场景分层抽样的，覆盖 API 类、故障类、功能细节类、版本类，同时刻意保留了百分之十到二十的冷门问题样本。每条     
  query 是用户在群里的真实问题，answer 标注的是对应的正确 chunk ID。我们把 384 和 512 两个配置跑检索评测。评测指标没有统一定一个，而是按问题类型拆成两组：单答案型问题用 MRR 做主页表、Recall@1 做辅助，综合推理型问题用 NDCG@5 做主页表、Recall@5 做辅助。最终 384 在这两组指标上都表现更好，同时 token 量更少、检索效率更高，所以我们选了 384。

  关于 overlap，因为我们按标题和段落做语义切分，不是按固定长度硬切，截断发生的比例非常低，所以 overlap
  设的是零。如果后续新增文档导致截断率上升，我们会考虑用父子 chunk 的方案来补救：子 chunk 命中时，把对应的父 chunk
  一并返回给大模型，用更长的上下文弥补语义断裂。

  评估这一步，我们实际没有跑 bootstrap 显著性检验，因为业务规模不大，评测集的差异 case 我们可以逐条人工检查来判断是不是噪声。而且线上我们有反馈 
  闭环——用户踩的答案、以及漏到人工群里的问题都会被日志记录下来，定期回流扩充评测集。如果未来数据量更大、评测成本更高，我们会加上 bootstrap      
  来避免噪声误导决策。

  关于表格的检索召回率，目前我们没有单独量化，因为在实际业务中，用户的问题主要以功能和故障描述为主，纯表格类问题占比较低。但这个判断更多是基于经
  验而不是数据验证，这一点是我需要改进的。


**Q2.** 你文档的格式是什么？纯文本、Markdown、PDF、Word、Confluence？不同格式你怎么做解析和归一化的？表格和图片你是怎么处理的？表格被切碎了怎么办？

 我们的 500 个文档全部来源于内部 Wiki 平台，原生 Markdown 格式，由团队自己维护在 Git 仓库中，不存在从 Confluence、Word
  或网页转换的场景。这保证了格式一致性，不存在转换过程中的编码错误或层级丢失。
  
  解析方案上，我们使用 LangChain 的 MarkdownHeaderTextSplitter。它底层基于 Python markdown 库做 AST
  解析，按标题层级递归拆分，能正确区分代码块内的 # 符号和真正的 Markdown 标题，不会出现正则匹配常见的误判问题。我们在选型时全量跑了 500
  文档的解析验证，检查了标题层级连续性、空 chunk 率和 chunk
  边界合理性，没有出现解析失败或静默切错的情况。坦白说我没有逐行读过它的源码，但通过输出验证确认了它在我们的文档集上是可靠的。

  表格、JSON 和代码块的处理，我们不做硬截断，而是全文保留，同时在离线清洗阶段用 DeepSeek V4 Flash
  模型为这些特殊文本块生成自然语言摘要。具体做法是 few-shot prompt，把上下文和特殊文本一起给模型，让它生成一句功能性的标题，比如一个 JSON       
  配置块会被总结为"region 白名单配置"。整个文档集大概有三十到四十个表格、不到五十个 JSON 块、三四十个代码块，每次调用的输入 token
  大约一千左右，输出只有几十个 token。因为我们用的是内部统一 API，成本在整体预算中几乎忽略不计。

  图片的处理是我们第一期明确做的取舍。目前模型不支持多模态，图片在解析阶段直接丢弃，在原文位置插入占位符标记。文档中与图片关联的引用文字如"如下 
  图所示"，当前没有做特殊清理，可能在个别 chunk 中遗留无意义的上下文断裂。但我们评估过我们文档的特点——内部文档以
  FAQ、功能说明和排障手册为主，不像论文或技术博客那样重度依赖图片传达关键信息，所以第一期没有投入工程资源处理这个问题。这是技术债，我记录了。   

  如果未来要支持图片，我会走图片转文本的路线，用视觉模型为图片生成文字描述嵌入原 chunk，继续复用现有的纯文本检索架构，不改检索管线。多模态      
  embedding 单独建索引的方案工程成本更高，在我们的场景里现阶段不划算。

**Q3.** 切片大小设了多少？这个数字是怎么定出来的？你有做过消融实验（ablation study）吗？chunk overlap 设了多少？overlap 设成多少对最终效果的影响你量化了吗？

我们的 chunk size 最终选了 384 token，overlap 设的是 0。

  选型流程分四步。

  第一步，确定候选值。我把 500 个文档全量跑了一遍 tokenizer，看了 chunk 长度的实际分布，中位数大概在 300 token 左右，所以定了 256、384、512     
  三个候选值在真实分布的两侧，覆盖了"偏小、适中、偏大"三个区间。

  第二步，截断率筛下限。在全量 500 文档上分别用三个上限做语义切分，统计截断率。截断的正确定义是：chunk 长度触达上限且与相邻 chunk
  共享标题路径。256 的截断率明显偏高，说明 256 装不下我们的内容，频繁腰斩语义，淘汰。384 和 512
  截断率都在百分之一左右，说明两个都足够大，截断率到这里就完成了它的使命，不再参与后续决策。

  第三步，检索评测做精选。384 和 512 两个配置全部跑检索评测。我们构造了 100 多条 QA
  pairs，按文档类型和问题场景分层抽样，高频约占八成，冷门约占两成。每条 query 标注了正确的 chunk ID 作为 ground truth。两个配置分别计算每条     
  query 的 MRR/Recall@1 或 NDCG@5/Recall@5 拆组评测。最终 384 的得分表现更好，差距不大。严格来说，我应该跑 bootstrap
  显著性检验来确认这个差距不是噪声——从 100 条里有放回抽样 10000 次，看差值分布，确认 p 小于 0.05。如果检验结果不显著，那就说明 384 和 512       
  在检索效果上无差异，此时选 384 是因为 token 更省、向量维度下的检索效率更高。好在我们业务体量小，实际中我们额外做了人工确认：把两个配置的差异  
  case 逐条拿出来看，判断 384 的检索结果是否确实更好，辅助验证了这个选择。

  第四步，chunk 质量人工抽查。切分完成后，我们对 384 配置下的 chunk 做了抽样检查，看了几十个 chunk，确认没有语义被明显截断或一个 chunk
  内堆叠多个不相关话题的情况。坦白说这一步是人工的，没有双人交叉验证，不够严谨。

  关于消融实验，我坦诚说没有做。我无法量化 header 拼接、特殊文本 LLM 摘要、chunk size 这三个设计各自对最终效果的贡献。这是我的一个盲区——如果面试
  官追问我哪个设计贡献最大，我答不出来确切的数字。正确的做法是把每个设计当开关，逐个关掉跑评测，看 MRR 掉多少。

  关于 overlap，我设的是 0。理由是我们的切分是按 Markdown
  标题层级加段落行做语义切分，不是固定长度硬切，截断率在百分之一以下，语义断裂的场景很少。所以 overlap
  用于补救截断的需求不存在。但我必须承认我没有验证 overlap 的另一个潜在价值——同一段内容出现在相邻两个 chunk
  里，检索时有双保险、命中概率可能更高。我没有跑过 overlap=0 和 overlap=50 的对照检索实验，所以我只能说我选了 0
  且目前线上检索效果可以接受，但我不能声称 overlap=0 一定是最优解。如果后续线上出现检索遗漏的 bad case，我会优先排查是否有语义刚好落在 chunk    
  边界导致向量相似度不够的情况，届时补做 overlap 的消融实验。

**Q4.** 你怎么处理文档更新的？文档有变更（新增、修改、删除）时，索引怎么同步？是全量重建还是增量更新？增量更新时怎么保证 chunk 之间引用关系不破裂？

 我们在每个 chunk 的 metadata 里记录了三个关键字段：file_name，作为文档级别的唯一标识；chunk_index，记录 chunk
  在文档内的切分顺序；chunk_hash，记录当前 chunk 内容的 MD5，用于变更检测。

  新增：用 file_name 查询向量库。查不到，说明是新文档，直接切分入库。查得到，返回一个二次确认提示——"该文件名已存在，确认覆盖？"操作人确认后才进 
  入修改流程。这个兜底避免了有人传了同名不同内容的文件导致旧文档被静默覆盖。

  修改：用 file_name 查出该文档的所有旧 chunk，按 chunk_index 排序。新文档切分后也按 chunk_index 排序。两个列表对齐逐个比对 chunk_hash。hash    
  相同的跳过不动，hash 不同的先删旧 chunk 再插新 chunk，旧文件中多出来的 chunk_index
  直接删除，新文件中多出来的直接插入。这样文档改了一个标点，只影响那一个 chunk，其余不动，不是全量删了重建。

  删除：用 file_name 查出该文档下所有 chunk，逐个删除后验证数量归零。

  并发安全：修改流程里删和插不是原子的——如果两个操作之间服务挂了，文档就丢了。当前用 Redis 分布式锁保护，锁粒度是 file_name
  级别，同一文档的并发操作被串行化，不同文档之间不互斥。锁有 30 秒过期兜底，防止死锁。

  索引同步：ChromaDB 的向量索引是增量更新的，增删改后自动同步。BM25 全文索引每次数据变更后从内存重建，规模在我们两千条 chunk
  级别下在毫秒级完成，不是瓶颈。

  chunk 引用关系：当前没有。因为按标题语义切分后截断率很低，chunk 是独立的检索单元，不需要父子或相邻引用。如果未来文档复杂度上升需要这种机制，在
   metadata 里增加 parent_chunk_id 字段，检索时命中子 chunk 一并带出父 chunk 即可。

  关于 file_name 做主键的局限：如果文件名被改了，旧 file_name 的 chunk 会变成孤岛数据残留库中，新 file_name
  走新增分支，可能引入重复内容。我们内部通过上传规范约束文件名唯一、不可随意更改，并在上传时做二次确认兜底。更严谨的做法应该用 Wiki
  页面路径或平台提供的稳定 ID 做主键，这个是未来的改进方向。

**Q5.** 有没有做文档的预处理或清洗？比如去掉页眉页脚、去除水印、处理乱码、格式转换错误？你是怎么发现这些脏数据问题的？

  我们目前没有做系统化的预处理管线，原因和局限我分开说。

  先说做过的部分。对于特殊文本块——表格、JSON 配置、代码块——我们在离线阶段用 DeepSeek V4 Flash 做 few-shot
  摘要生成，把文本块和它周围的上下文一起给模型，让它输出一个功能性的标题，比如"region 白名单配置"或"API 错误码对照表"，这个标题作为 chunk
  的前置信息打入 content，补充语义。这个处理是规则发现特殊文本后用正则定位，然后离线批量调用 LLM 完成。

  图片我们做了基本处理：解析时检测到图片，替换为占位符标记，同时在 prompt
  中告诉生成模型"当前文档中图片已被移除，如遇到'如下图'等无意义引用请忽略"。

  坦白说下面几个点是我没有做但应该做的。

  第一，文档质量本身没有审查。500
  个文档由不同人不同时期编写，风格和粒度不一致。有的可能是详细的操作步骤，有的可能就一句话。没有经过质量筛选就直接入库了，这可能导致 RAG        
  召回了一个写得模糊的文档，LLM 基于它生成了同样模糊甚至错误的答案。正确做法应该在上线前对入库文档做一轮人工审核或 LLM
  质量打分，标出低质量文档，要么优化要么暂不索引。

  第二，图片的类型和分布我没有统计。我只知道图片数量不多，但我不知道其中多少是截图（报错、配置界面），多少是架构图、流程图。这两类图片的处理完全
  不一样：截图的 OCR 文字提取是有用的，但架构图里语义在拓扑关系里，纯 OCR
  提取出来是一堆散落的名词，没有意义。我把所有图片归成一类说"不重要"，这是拍脑袋，不是评估。正确做法应该抽样统计图片类型分布后再决定方案。      

  第三，图片引用文字的清理是没做的。占位符能标记图片丢了，但不能修复上下文断裂。"按照下图红色框所示，点击提交按钮"——图片没了，这句话对 LLM      
  就是噪声。如果说"图片数量不多所以没处理"，但面试官会反问"那你验证过吗？"我没有验证，这是我的盲区。

  如果后续要做图片支持，我的方案是：先抽样统计图片类型比例。如果截图为主，走 OCR 文字提取 +
  嵌入原文，不改检索架构。如果架构图/流程图占比高，考虑用视觉模型生成自然语言描述替代图片，仍然走纯文本检索。多模态 embedding
  独立建索引的方案工程量大，只有在图片成为核心信息载体时才值得投入。就我们内部 FAQ
  和排障手册的文档特点判断，大概率走第一条路就够了，但这个判断目前缺乏数据支撑。

### 1.2 Embedding & 向量化

**Q6.** Embedding 模型选的是什么？为什么选这个？你对比过哪些模型？在你们自己的数据集上做评测了吗？评测指标是什么？
 我们用的是阿里 text-embedding-v3，内部私有化部署，不走公网
  API。这个选择的背景是信息安全合规要求——所有模型必须部署在内网，网络完全隔离，不能用外部
  API。因此我们的候选池天然受限，只能在内部已部署的模型里选，当时可选的是 V2 和 V3 两个版本。

  选型评测的做法是：在 100 条 QA pairs 上跑检索，但评测指标不是统一的。我把评测集按问题类型拆成了两组。单答案型问题——比如
  FAQ、配置项查询，命中一个 chunk 就能回答——用 MRR 做主页表、Recall@1 做辅助，因为 MRR 衡量的是第一个相关 chunk
  的排名，刚好匹配这类场景的需求。综合推理型问题——比如故障排查、跨文档分析，需要多个 chunk 共同支撑答案——用 NDCG@5 做主页表、Recall@5
  做辅助，因为 MRR 在多相关 chunk 场景只看第一个的排名，完全忽略后面几个相关 chunk 的排序质量，而 NDCG 把所有相关 chunk
  的排名都计入，只是权重递减，更能反映综合推理型场景的实际检索质量。

  评测结果 V3 在这两组指标上都优于 V2，所以选了 V3。坦诚说我们没有跑 bootstrap 显著性检验，因为评测集规模不大，差异 case
  我们逐条人工分析过，能确认提升是真实改善而非噪声。如果评测集扩展到几百条以上，应该加上显著性检验。

  关于维度选择，V3 支持多个输出维度。我把 512、768、1024 都跑了一遍评测，找到了边际拐点——从 512 到 768 有明显提升，从 768 到 1024
  几乎持平但存储和检索延迟多了不少，所以选了性价比最高的那个。不能无脑上最大维度，维度是存储成本和检索精度的 tradeoff。

  候选池局限于阿里两个模型，我清楚——如果有条件扩展，BGE 的中文模型和 Jina
  的多语言模型在公开评测上对中文支持也很好，但受合规约束目前没有实测，这是环境限制不是技术决策。

  关于模型升级的规划，embedding 模型一旦切换、维度一旦变化，必须全量重建索引。我设计的升级流程是：旧版本索引继续服务线上，同时在后台全量重建新版
  本索引，两个索引并行跑一段时间，对同一批 query 比较检索结果，确认不退化后再切流量，旧索引保留一周做回滚兜底。当前规模下全量重建成本在百万     
  token 级别，完全可以接受。


**Q7.** 你用的是什么向量数据库？Milvus、Pinecone、Weaviate、Qdrant 还是自研？为什么选它？跟你们现有基础设施怎么配合的？
我们用的是 ChromaDB，选型逻辑不是因为它"刚好有"，而是按综合代价公式做了六个方案的对比后，它在我们场景下代价最低。

  综合代价 = 部署成本 + 运维成本 + 学习成本 + 切换成本 + 性能折损。

  Elasticsearch。如果团队已有 ES 集群，在 dense_vector 上加 HNSW 索引是最省事的方案——一个引擎同时提供
  BM25、向量检索、混合打分，高可用自带，中文分词现成。但我们当前环境没有 ES 集群，引入 ES 意味着引入 JVM 运维、heap 调优、版本升级时的 breaking 
  change，而团队主技术栈是 Python，出了问题排查成本很高。引入一个新中间件的代价远超我们的需求。

  Milvus。十亿级向量的最佳方案，但完整部署依赖 etcd、MinIO、Pulsar，运维四个中间件来完成两千个 chunk 的检索是杀鸡用牛刀。Milvus Lite 才是       
  ChromaDB 真正的竞品，pip install 即可运行，底层也是 hnswlib。坦诚说我没有做 Milvus Lite 和 ChromaDB 的同等数据量对比
  benchmark，这是选型流程上的缺失。

  Qdrant。Rust 实现，单机吞吐量同类领先。但它是独立进程的 Rust 二进制，部署多一个环节。如果出 bug，翻 Rust 源码的门槛我够不着。在两千条 chunk   
  的规模下，hnswlib 和 Qdrant 的性能差异根本感知不到，但运维复杂度的差异是实打实的。

  Weaviate。自带 vectorizer、generative、rerank 模块，RAG 全家桶。但我们架构是检索和生成分离——embedding 自己调、LLM 自己调、rerank
  自己调。Weaviate 的内置模块在我们架构里全是冗余，反而引入了额外的模块启用禁用配置。ChromaDB 只做"存向量加查向量"一件事，和我们自己控制        
  pipeline 的设计一致。

  FAISS。Facebook 的向量检索库，算法丰富、GPU 加速成熟。但它是库不是数据库——没有持久化、没有元数据管理、没有增删改 API。要用 FAISS 做 RAG       
  必须自己在上层实现所有这些，相当于把 ChromaDB 封装好的功能重写一遍。FAISS
  适合需要自定义索引策略的场景，对我们这种需要"能存能查"的需求来说太底层了。

  pgvector。如果团队已有 PostgreSQL，一个 CREATE EXTENSION 就搞定，向量和业务数据在一个事务里，ACID 保证。但当前环境没有 PG 实例。pgvector 的   
  HNSW 索引构建受 PostgreSQL WAL 写入路径影响，比 ChromaDB 慢。如果未来需要向量加元数据混合过滤，pgvector 是优势，但当前没有这个需求。

  最终选 ChromaDB 的核心原因：部署就是 pip install，零外部依赖；嵌入在 Python 进程内，没有独立服务需要监控重启；三个 API 解决所有操作；百级     
  chunk 下 HNSW 默认参数的召回率和延迟完全够用；未来如果换引擎，只需要改 VectorStore 这一个适配层。

  ChromaDB 的明确局限我也清楚。第一，单机部署不支持分布式，数据量到十万级 chunk 时索引内存会到几十
  GB，单机扛不住，必须换引擎。第二，没有内置高可用，进程崩了检索链路就中断，没有主从或副本自动切换。第三，它的 where
  过滤是向量检索之后的后过滤，不是索引阶段的前置裁剪——如果你需要按模块或时间范围过滤，白白检索了全量再扔掉大部分，目前我们没有复杂过滤需求所以不
  受影响。第四，持久化依赖本地磁盘的 SQLite 加二进制文件，没有备份机制。当前两百多条 chunk
  的重建成本不到半分钟，所以没做冷备，但如果规模到万级，必须规划定期备份。


**Q8.** 向量检索走的是哪个相似度度量？cosine、dot product 还是 Euclidean？为什么？你做 A/B 测试验证了吗？
 阿里 text-embedding-v3 模型训练时用的是 cosine 相似度，也就是输出的向量是在单位超球面上的，语义由方向决定而非向量长度。所以我 ChromaDB 的     
  collection 里显式指定的是 cosine，和 embedding 模型的训练目标对齐。如果用 L2 且 embedding
  没有做归一化，不同文档向量的模长差异会污染排序结果——L2 比的是坐标绝对值，不关心语义方向。一个长的无关文档和一个短的匹配文档，L2
  可能把长的排前面。

  但我坦诚说，我没有验证过 embedding 模型返回的向量是否确实是归一化过的，即向量模长是否为 1。如果归一化了，cosine 和 L2
  在数学上等价，排名结果一样。如果没归一化且 chunk 长度差异大，检索就一直有偏差。目前我的 collection 已经显式设为
  cosine，从根源上消除模长污染的潜在风险，但最终验证这步我没有做，应该补上。


### 1.3 索引 & 检索策略

**Q9.** 你是不是只做了向量检索？有没有做混合检索（hybrid search）？比如 BM25 + 向量？如果有，怎么融合两个通路的分数的？用的是 RRF（Reciprocal Rank Fusion）还是线性加权？融合参数怎么调的？

我们没有只做向量检索，做的是向量加 BM25 双路混合检索。整个检索链路的入口是先做 query 改写，再走双路检索，最后 RRF 融合。

整个流程是：用户问题进来后，先经过 query 改写步骤。改写器做两件事：一是用 LLM 将用户的口语化表述改写为文档风格的专业表述，比如把"手机收不到短信"改写为"SMS通知渠道未收到告警"；二是通过一个动态映射表，将用户常用的非专业词汇替换为系统内的专业术语，比如"OA"映射为"优化顾问"、"那个监控"映射为"OA监控告警"。映射表存储在动态配置中，可以在运行时直接修改，不需要重启服务或重新部署。映射表的维护来源是线上 bad case 复盘——用户点击"没用"的反馈会被记录下来，包括原始 query、改写后的 query、以及最终检索结果。在项目上线后的前一个月，我们每天都会看这些 bad case 日志，分析是改写步骤出了问题还是检索步骤出了问题。如果改写出了问题，就更新映射表；如果是检索出了问题，就排查是 jieba 分词错误还是 embedding 召回失败。

改写之后，query 并行进入两条检索通路。

向量路：用阿里 text-embedding-v3 对改写后的 query 做 embedding，进 ChromaDB 的 HNSW 索引做余弦相似度检索，返回 top-k 个候选。向量检索擅长语义相似和同义改写，但对精确关键词、专有名词、缩略语的匹配能力弱。

BM25 路：走 ChromaDB 内置的 BM25 做关键词检索，同样返回 top-k 个候选。BM25 擅长精确关键词匹配，正好弥补向量路在专有名词和缩写上的盲区。分词器用的是 jieba，因为我们的文档是中文技术文档，有大量内部术语，默认词库切不准。我们维护了一套自定义词库，包含业务专有名词和缩写，比如"优化顾问"、"region白名单"、"多AZ"、"SMS通知"等。词库同样支持运行时热加载，发现新术语后添加到词库文件，调 reload 接口即刻生效。词库的维护来源和映射表一样，来自线上 bad case 复盘——如果发现是因为分词错误导致 BM25 路失效，就加词库。

两路结果通过 RRF（Reciprocal Rank Fusion）融合成最终候选列表。选 RRF 而不是线性加权的原因是：向量检索的余弦相似度和 BM25 的 TF-IDF 分数量纲完全不同，直接线性加权会有一路分数被另一路淹没。RRF 只关心排序位置，天然规避了量纲问题。公式是 score = alpha/(k+rank_vec) + beta/(k+rank_bm25)，k 取 60。

alpha 和 beta 当前都设的是 0.5。坦诚说这个值是初始默认值，没有做过系统化的网格搜索调参。正确的做法应该是基于已有的 100 条 QA pairs，alpha 从 0 到 1 以 0.1 为步长切 11 组，每组跑一遍 RRF 融合后算指标——两路的排序结果早就有了，改 alpha/beta 只是重新算融合分数，不需要重跑检索，一秒算完。我没做是因为上线初期效果已经满足预期，就搁置了。这是技术债。

关于要不要加第三路，我评估过 HyDE 和 Small-to-Big。HyDE 的核心思想是用 LLM 生成假设答案，把"问题 vs 答案"的匹配变成"答案 vs 答案"的匹配。经典 HyDE 不适合私有知识场景，因为 LLM 不了解内部文档，凭空生成的假设答案用词对不上。但有一个变体可行：先跑一轮快速向量检索取 top-3，把真实 chunk 给 LLM 让它基于真实内容生成假设答案，再做第二轮检索——这样风格和术语都来自真实文档。我没有实施，因为当前双路效果已满足指标，且我们的改写器已经在做风格对齐的工作。Small-to-Big 是返回时把命中小 chunk 所属的父文档一并返回，不改检索逻辑所以不提升 MRR，但提升答案完整度。没有实施是因为当前未发现因上下文不足导致的生成 bad case。两路到三路有边际收益，但三路以上收益递减明显，双路是当前业务的最优平衡。

评测指标方面，我们的 100 条 QA pairs 评测集按问题类型拆成两组：单答案型（命中一个 chunk 就能回答）用 MRR 做主页表、Recall@1 做辅助；综合推理型（需要多个 chunk 共同支撑）用 NDCG@5 做主页表、Recall@5 做辅助。MRR 只看第一个相关 chunk 的排名，在多相关 chunk 场景会低估排序缺陷。Recall@5 只看有没有命中、不关心排序。NDCG 把所有相关 chunk 的排名都计入，排越前权重越大，能完整衡量多相关 chunk 场景的整体排序质量。
