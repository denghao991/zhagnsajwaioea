# Chunk Size 选型方法论

## 一、什么是 Chunk Size

将长文档切分成固定 token 数上限的语义块，每个块是检索的最小单元。chunk_size 决定了块的最大容量，chunk_overlap 控制块间重叠量。

我的分块器是 MarkdownChunker：先用 LangChain MarkdownHeaderTextSplitter 按 `#` / `##` 标题做语义边界切分，再用 RecursiveCharacterTextSplitter 按段落累积到 chunk_size 上限。这意味着块边界由文档标题结构主导，chunk_size 只是一个"容量上限"，不是"固定切分点"。

## 二、评估指标体系

### 静态指标（不需要跑 MRR，算 token 数即可）

| 指标 | 公式/定义 | 告诉什么 | 决策价值 |
|------|-----------|---------|---------|
| **P50/P75/P90** | 所有块 token 数的百分位数 | P75=800 表示 75% 的块 ≤800 token | P75 接近 chunk_size 时说明约 25% 的块被硬截断 |
| **碎片率** | 信息密度极低的块（如只有标题的块）占比 | 分块器是否过度切碎文档结构 | 碎片率从高位跳变到接近 0% 的拐点 = chunk_size 软下限 |
| **填充率** | 平均 token 数 / chunk_size | 分块器对容量的利用效率 | 过低说明 chunk_size 选大了，余量被浪费 |

**关于碎片率 5% 阈值的说明**：5% 本身是一个工程锚点（20 个块里有 1 个碎片），但方法论的核心不是这个具体数字——碎片率在 chunk_size 过小时是**跳变**而非渐变（标题被突然孤立），拐点位置对 3%/5%/7% 都不敏感。如果面试官追问，可以说明关键在"拐点"不在"阈值"，同时用实际数据展示跳变幅度。

**关于 P75 ≥ chunk_size × 0.9 阈值的说明**：这个阈值来自 MarkdownChunker 的设计前提。当 25% 的块触达容量上限时，意味着 token 计数已经取代标题结构成为切分的主导因素——chunker 的"语义边界优先"设计意图正在失效。这与其说是"硬上限"，不如说是"递增收益消失点"（diminishing return point）：继续增大 chunk_size 对已充分容纳的短节无益，却会让更多长节被容量限制切碎。

### 动态指标（需要跑 QA 评估）

**MRR（Mean Reciprocal Rank）**：

$$\text{MRR} = \frac{1}{N} \sum_{i=1}^{N} \frac{1}{\text{rank}_i}$$

- 关注**第一个正确答案排在第几位**
- 对排名顺序极其敏感：排第 1 得 1 分，排第 3 只得 0.33 分
- 适合 RAG 场景，因为最终喂给 LLM 的窗口有限，排前面的块最有价值

**Recall@K**：

$$\text{Recall@K} = \frac{1}{N} \sum_{i=1}^{N} \frac{|\text{正确答案} \cap \text{TopK}|}{|\text{正确答案}|}$$

- 关注**前 K 个结果覆盖了多少比例的正确答案**
- 对排名不敏感，排第 1 和第 5 一样算命中
- 当每个 query 只有 1 个正确答案时，退化为命中率

两者都是 N 条 query 的平均值。在 chunk_size 选型中 MRR 比 Recall@K 更敏感——两个不同 chunk_size 可能 Recall@5 都是 1.0（都没漏），但 MRR 能区分谁排得更靠前。

**MRR 的局限性**：对多跳推理（multi-hop）场景不敏感，因为它只看第一个正确答案。如果系统需要跨块综合推理，应补充 Recall@K 或多答案 MRR 变体。选型方法论本身不受此限制——将 MRR 替换为其他下游指标，流程不变。

## 三、选型方法论：双向逼近

不要拍脑袋选 6-8 个值跑，也不要在单个方向上盲目扫描。用**双向逼近**将搜索空间从无边界穷举收敛到有边界定向评估。

### 前提：固定控制变量

在开始选型前锁定以下变量：
- **chunk_overlap**：固定在 chunk_size 的 10-15%（如 chunk_size=512 时 overlap=64），后续如需微调再做敏感性分析
- **embedding 模型**：固定为当前使用的模型（方法论的流程对任意 embedding 模型通用，但具体最优值会随模型变化——这是"方法与参数分离"的优势，不是缺陷）
- **检索管线**：RRF k 值、top-K、rerank 配置保持不变
- **查询集**：至少 30 条，覆盖三类查询——事实定位查询（"X 是什么"）、跨文档综合查询（"A 和 B 什么关系"）、长尾细节查询（"某段提到的具体参数值"）

同时锁定控制变量不是为了回避它们的交互效应，而是工程上必须收敛搜索空间。如果对所有变量做全因子设计（chunk_size × overlap × embedding × k × ...），组合爆炸不可承受。

### 第一步：分析文档结构

统计你的文档集：
- 每个 `##` 标题下平均有多少 token？（记作 $T_{avg}$）
- P90 token 数？（$T_{p90}$）
- 是否存在极端长节（单个 `##` 下超过 2000 token）需单独标记

围绕 $T_{avg}$ 选 3 个候选值：一个偏小、一个接近、一个偏大。例如 $T_{avg} \approx 380$，候选取 `[256, 384, 512]`。

**$T_{avg}$ 方差的处理**：如果文档分布极度偏斜（存在一个 5000 token 的超长节拉高了 $T_{avg}$），不必担心——P75 会在第二步自动把上限往下压，因为那些超长节会导致大量块触达容量上限。极端文档结构不是这个方法的盲区，它会被静态指标自动捕获。

### 第二步：定上限（从大到小，用 P75）

从最大候选（如 1024）往下扫，找到 **P75 ≥ chunk_size × 0.9** 的点。超过这个值，语义边界被容量压制，继续增大 chunk_size 的收益递减——这是"递增收益消失点"。

### 第三步：定下限（从小到大，用碎片率）

从 256 往上扫，找到**碎片率从高位跳变到接近 0% 的拐点**。低于这个值，标题结构被过度打碎，分块器在按 token 计数切而不是按语义结构切。

### 第四步：区间内 MRR 定标

在上下限之间取 2-3 个候选值跑 MRR。

**统计显著性的处理**：
- 对 MRR 结果做 bootstrap 重采样（1000 次），计算 95% 置信区间
- 如果候选间 MRR 差异落在置信区间重叠范围内（差异不显著），选较大的 chunk_size——理由：大块提供更完整的上下文给 LLM 生成阶段，且不会损害检索质量（因为检索质量在候选区间内持平）
- 如果某个候选的 MRR 显著优于其他，选它
- 如果最优候选恰好落在上下限边界上，往该方向补测一个候选值确认

## 四、三个指标的角色

```
        碎片率（下界）           P75（上界）
            |                     |
            v                     v
    不要切太碎              不要切太狠
   （尊重文档结构）        （避免硬截断）
            |                     |
            +--------+-----------+
                     |
                  MRR 区间
              （最终裁决谁最优）
```

- **碎片率**回答：文档结构有没有被尊重？（下界问题，定下限）
- **P75** 回答：截断有没有发生？（上界问题，定上限）
- **MRR** 回答：检索有没有受益？（最终裁决）

碎片率和 P75 是正交的——一个看下界，一个看上界。如果在 top-down 单向评估中碎片率为 0，不代表它没价值，只是你的 chunk_size 太大把它掩盖了。双向扫描才能让两个指标都发挥作用。

**静态指标和动态指标的角色分工**：碎片率和 P75 的作用是**排除不可接受的选择**——chunk_size 太小导致结构碎裂、太大导致结构被容量压制。它们测量的是分块器本身的行为，而非下游检索表现，因此能在不跑昂贵 MRR 评估的前提下缩小搜索范围。区间确定后，我们信任 MRR 在这个小区间内能反映真实的检索质量差异。

## 五、常见误区

1. **chunk_size 必须是 2 的次幂**：不是。128/256/512 只是看起来整齐，换成 150/300/600 效果一样。2 的次幂是计算机内存传统，跟 NLP 无关。

2. **跑 6-8 个候选值逐个比较 MRR**：浪费。MarkdownChunker 的块边界由标题结构主导，chunk_size 增大到超过 $T_{p90}$ 后块结构不再变化。3 个候选在大多数情况足够。

3. **碎片率低 = 分块质量好**：不一定。chunk_size=2048 时碎片率可以是 0%，但 5 个不相关的段落可能被塞进同一个块，造成语义混淆——只不过碎片率测不出这个问题。这也解释了为什么碎片率只能定下限、不能选最优。

4. **Recall@K 高就可以**：不够。Recall@5=1.0 只说明正确答案没漏，但 MRR 可能只有 0.3（正确答案被噪声挤到了第 4 位）。RAG 场景下排名顺序比覆盖度更重要。

5. **最优 chunk_size 对所有 embedding 模型通用**：不是。不同 embedding 模型对语义窗口的敏感度不同（如 text-embedding-v2 偏好 512 附近的块，BGE-M3 可能在 256-384 表现更好）。换模型需要重新跑数据，但方法论的**流程**不变——这是方法与参数分离的优势。

## 六、实际项目经验

在 tiny-rag 项目中，对 13 篇中文技术文档测了 128/256/384/512/768/1024 共 6 个 chunk_size：

- 静态指标：128 填充率最高（79.7%），碎片率在 128 时显著升高
- 结合 RRF 混合检索（alpha=7/beta=3），最优 MRR = 0.7234

如果按双向逼近方法论重做，候选从 6 个收敛到 2-3 个：先算 $T_{avg}$ → 定 3 个候选 → P75 定上限 → 碎片率定下限 → MRR 定标。

## 七、面试追问预案

以下是面试官最可能追问的点及答辩策略：

### Q1："你的查询集怎么构造的？有统计显著性检验吗？"

**答辩**：承认当前查询集规模（13 条）有限，这是评估成本约束下的工程权衡。改进方案明确：
- 扩展到 30-40 条，覆盖三类查询：事实定位（单文档单段）、跨文档综合推理、长尾细节
- 加入 bootstrap 置信区间，汇报 MRR 差异的显著性水平
- 小样本下，"差异不显著则选较大的"是一个可辩护的保守策略

### Q2："换一个 embedding 模型，你的结论还成立吗？"

**答辩**：具体最优值会变，方法论流程不变。方法与参数是解耦的——同样的四步流程可以对 text-embedding-v3、BGE-M3、OpenAI embedding 重新跑一遍，得出各自的最优值。这恰恰是方法论的价值：你不是在记一个固定答案，而是在掌握一套可迁移的选型流程。可以展开讨论 embedding 模型特性如何影响对 chunk_size 的敏感度——高维模型（1024d+）通常对块大小更鲁棒，低维模型更敏感。

### Q3："碎片率 5% 和 P75 0.9 这两个阈值是拍脑袋的吗？"

**答辩**：分解为两部分——

碎片率：5% 是工程锚点，但方法论依赖的是**拐点**而非具体阈值。碎片率在 chunk_size 过小时是跳变（从 0% 跳到 10-20%），拐点位置对 3%/5%/7% 不敏感。如果面试官需要，可以用实际数据演示跳变幅度。

P75 0.9：从 MarkdownChunker 的设计前提推导——当语义边界被容量上限压过时，chunker 的设计意图失效。这是"递增收益消失点"而非绝对禁入线。

### Q4："你怎么知道最优 chunk_size 不依赖于你选的 overlap？"

**答辩**：承认 chunk_size 和 overlap 存在交互效应，但全因子设计的组合爆炸在工程上不可行。先固定 overlap 在标准比例（10-15%），选好 chunk_size，再回过头做 overlap 的敏感性分析——这是工程上合理的主次顺序。

### Q5："如果你选的 3 个候选 MRR 全都不显著，怎么办？"

**答辩**：执行 fallback 规则——选较大的 chunk_size。理由：大块提供更完整的上下文给 LLM 生成阶段，同时不会损害检索质量（因为 MRR 在候选区间内持平）。这个规则有明确的逻辑支撑，不是随机选择。如果时间允许，可以在更大范围内（如 5 个候选）重新验证。
