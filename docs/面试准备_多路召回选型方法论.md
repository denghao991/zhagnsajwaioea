# 多路召回选型方法论

## 一、为什么 embedding 之后要选召回路径

chunk_size 定了粒度，embedding 定了向量路的质量。但**单一向量检索有天然盲区**——用户 query 和文档的语言风格不同，短 query 语义稀疏，长 query 含噪声，专有名词和缩写 embedding 模型没见过。这些场景下单靠向量检索会丢答案。

多路召回就是让不同的"搜索引擎"各自发挥所长，然后融合互补。各路的侧重不同：

| 召回路径 | 擅长 | 盲区 |
|---------|------|------|
| **向量检索（Dense）** | 语义相似、同义改写、跨语言 | 专有名词、缩写、精确匹配关键字 |
| **BM25（Sparse）** | 精确关键词、专有名词、缩写、数字 | 语义相似但用词不同、"说人话"和"书面语"的gap |
| **HyDE（生成式）** | query 语言风格和文档差异大的场景 | 依赖 LLM 质量，幻觉可能带偏检索方向 |
| **Small-to-Big（父文档）** | 小块检索精度 + 大块上下文完整度 | 不改检索质量本身，只改善返回给 LLM 的内容 |

**两路（向量+BM25）是基线，三路是甜点，四路以上边际收益递减。**

本项目当前是向量+BM25 双路。在 embedding 选型中已确认向量路 MRR 从 v2 的 0.66 跃升至 v3-512d 的 0.86。现在考虑加第三路（HyDE 或 Small-to-Big）是否能进一步提升，以及如何选。

## 二、两路基线：向量 + BM25

### 当前状态

```
question → query改写 → embed问题 → 向量检索top-k₁
                            → BM25检索top-k₂
                                       ↓
                              RRF融合(alpha/beta)
                                       ↓
                                  候选列表
```

### 已知评估数据

从 embedding 评估的 per-query 分析中可以看到，有两条 query 在当前管线（含改写 + 向量检索）下表现异常：

| Query | 向量检索 rank | 改写质量 | 根因 |
|-------|-------------|---------|------|
| "OA监控消息，我手机收不到告警短信..." | null（v2）/ 1（v3） | v2 改写为"优化" | 改写器不稳定导致向量路完全失效 |
| "OA应用场景是什么" | 8-13（所有模型） | 正常 | query 本身语义稀疏，和文档语言风格差异大 |

第一条 query 说明**改写器是单点故障**——改写崩了向量路直接废。第二条说明有些 query **天生不适合向量检索**（短 query 对长文档，语义交叠少），但 BM25 路如果用"应用场景"做关键词匹配恰好能救回来。

两路的互补性已经存在，但当前改写器的脆弱性是一个未解决的问题。这正是 HyDE 的切入点。

## 三、HyDE（Hypothetical Document Embeddings）：用假设答案代替 query

### 核心思想

```
传统流程:  query → embed → 检索
HyDE流程:  query → LLM生成假设答案 → embed假设答案 → 检索
```

普通检索是用"问题"去匹配"答案"——但问题和答案的语言风格完全不同。用户问"手机收不到短信"，文档写的是"SMS 通知渠道配置说明及故障排查"。向量空间里这两个相距很远。

HyDE 的做法：先让 LLM 根据 query 随便写一段假设的回答（不需要是真实答案，只需要语言风格像文档），然后用这段假设回答去检索。匹配模式从"问题 vs 答案"变成了"答案 vs 答案"，语义距离大幅缩短。

### 为什么比查询改写好

你的评估已经暴露了改写器的问题：14 条 query 中 2 条被改写器毁了（"优化"、"优化顾问(OA)"），改写器是非确定性的、不可控的。

| 维度 | 查询改写 | HyDE |
|------|---------|------|
| 输出 | 改写后的 query（仍是问题风格） | 假设答案（文档风格） |
| 失败模式 | 过度压缩、丢失关键词（如"优化"） | 幻觉内容，但风格仍然像文档 |
| 对检索的影响 | 改写崩了 → 向量路直接废 | 幻觉内容 → 仍能通过风格匹配找到近似文档 |
| 可控性 | LLM 温度、prompt | LLM 温度、prompt、可限定生成长度 |

HyDE 的容错性更强：即使 LLM 生成了不太准确的假设答案，只要语言风格和文档领域一致（技术文档的中文风格），就能把相关文档拉近向量空间。而改写器一旦把 20 个字的 query 压成 2 个字，向量路就彻底没法救了。

### 具体流程

```
query: "OA监控消息，我手机收不到告警短信"

Step 1: LLM生成假设答案 ↓

"OA监控告警通知通过短信渠道发送，如需接收告警短信请确认：
 1. 告警规则已配置且已启用
 2. 通知渠道已勾选'短信'选项
 3. 手机号码已正确填写"

Step 2: embed这段假设答案 → 得到向量

Step 3: 用这个向量去 ChromaDB 检索

Step 4: 返回的 top 结果比直接用 query embedding 更精准
       因为匹配的是"文档风格 vs 文档风格"
```

### 代价与权衡

| 维度 | 影响 |
|------|------|
| 延迟 | 多一次 LLM 调用（~200-500ms），但可与 BM25、原始向量检索并行 |
| 成本 | 每次 /ask 多一次 LLM token 消耗（生成 ~100 token，成本极低） |
| 存储 | 零额外存储（不建新索引，复用 ChromaDB） |
| 失败处理 | LLM 生成失败（超时/报错）→ 降级：跳过 HyDE 路，只用向量+BM25 双路 |

### 选型判断：什么场景下 HyDE 值得加

| 条件 | 满足？ | 说明 |
|------|-------|------|
| query 和文档语言风格差异大 | 是 | 用户口语化 query vs 技术文档 |
| 已有改写器且不稳定 | 是 | 14% 的 query 被改写破坏 |
| 能承受每 query 多一次 LLM 调用 | 是 | 内部知识库，并发低，延迟容忍度高 |
| 不需要额外基础设施 | 是 | 复用已有 LLM 和 ChromaDB |

四个条件全部满足，HyDE 是当前场景最值得加的一路。

## 四、Small-to-Big（父文档检索）：小块检索，大块返回

### 核心思想

```
索引阶段:  小块 embed + 检索（保持精度）
返回阶段:  返回小块所属的完整父文档（保证上下文完整）
```

chunk_size 选型中的核心矛盾：小 chunk 检索精度高（语义聚焦，不会被噪声拖累排名），但上下文不够（LLM 看不到完整段落难以生成好答案）；大 chunk 上下文完整但检索精度低（多个不相关句子挤在同一块，语义混淆）。

Small-to-Big 同时拿到两者的好处：用小块做向量检索（精度），返回时把小块所属的父文档（比如同一个 `##` 标题下的全部内容）一起返回给 LLM。

### 为什么你的 MarkdownChunker 天然支持

你的分块器已经按 `#`/`##` 标题切分，每个 chunk 天然知道自己属于哪个标题、属于哪个父段落。不需要重新设计索引结构——只需在生成阶段多返回一个 `parent_content` 字段。

```
chunk_size=384 小块:
┌────────────────────────────────┐
│ ## 监控告警配置                   │
│ 告警规则包括CPU、内存、磁盘...       │  ← chunk_id: "监控告警#0", tokens: 380
│ ...                              │
└────────────────────────────────┘
┌────────────────────────────────┐
│ ## 监控告警配置                   │
│ 配置完成后预计5分钟内生效...         │  ← chunk_id: "监控告警#1", tokens: 350
│ ...                              │
└────────────────────────────────┘

Small-to-Big 返回:
  检索命中 chunk "监控告警#1"
  → 返回 "监控告警#1" + 它的父文档（同一 ## 标题下的 "监控告警#0"+"监控告警#1" 合并）
  → LLM 看到的是完整上下文
```

### 两种实现方式

**方式 A：句子窗口检索（Sentence Window）**

```
每个 chunk 存的时候带上它前后各 N 个 chunk 的内容。
N=1 时相当于把 chunk[i-1] + chunk[i] + chunk[i+1] 拼接成一个窗口返回。
```

实现简单，但窗口是固定的（N=1），不能自适应文档结构。适合滑动窗口分块器（无标题结构）。

**方式 B：父文档检索（Parent Document）**

```
每个 chunk 存的时候记录它所属的父文档 ID。
返回时按父文档 ID 聚合所有子 chunk，拼接成完整上下文。
```

实现稍复杂，但窗口由文档标题结构定义，更自然。**适合 MarkdownChunker（你的场景）**。

| 维度 | 句子窗口 | 父文档检索 |
|------|---------|-----------|
| 窗口定义 | 固定 N | 标题结构 |
| 自适应 | 否 | 是（随 ## 段落长度变化） |
| 实现成本 | 低 | 中（需存 parent_id） |
| 适用分块器 | 滑动窗口 | 标题切分 |

### 代价与权衡

| 维度 | 影响 |
|------|------|
| 索引存储 | 需要额外存一个 `parent_id` 字段（每个 chunk 一个字符串，忽略不计） |
| 检索延迟 | 零额外延迟（检索逻辑不变，只是返回内容多了） |
| tok消耗 | LLM 输入 token 增加（返回父文档而非单个 chunk），生成阶段成本上升 |
| 适用条件 | 文档标题结构良好（你的文档是技术文档，结构好） |

### 选型判断：什么场景 Small-to-Big 值得加

| 条件 | 满足？ | 说明 |
|------|-------|------|
| 小块 chunk_size（< 512） | 是 | chunk_size 384，上下文有限 |
| 文档有清晰的层级结构 | 是 | 技术文档，## 标题结构良好 |
| 下游 LLM 需要完整段落才能生成好答案 | 是 | 客服类回答需要上下文背景 |
| 不改检索逻辑 | 是 | 只改返回内容，检索管线不变 |

Small-to-Big 不改检索质量本身——它不提升 MRR——但能提升 LLM 生成质量。评估它的方式不是看 MRR 变化（MRR 不变，因为检索步骤没变），而是看**用户对回答完整度的满意度**或**人工评估生成质量**。

## 五、HyDE vs Small-to-Big：该加哪一个

两者解决的是不同层面的问题，不互斥，可以都加。但出于评估成本考虑，需要排优先级。

| 维度 | HyDE | Small-to-Big |
|------|------|-------------|
| 改善什么 | **检索质量**（让 query 更好地匹配文档） | **生成质量**（让 LLM 看到更多上下文） |
| 评估指标 | MRR、Recall@K（可量化） | 生成质量人工评估（需标注） |
| 实现成本 | 多加一次 LLM 调用 | 一个 parent_id 字段 |
| 风险 | LLM 生成失败 → 降级双路 | 零风险（检索逻辑不变） |
| 增量价值（估计） | +5-10% MRR | 不改变 MRR，但显著提升答案完整度 |
| 对改写器问题的解决 | 替代改写器，消除单点故障 | 不解决改写器问题 |

**推荐优先级：HyDE > Small-to-Big**

理由：
1. HyDE 直接解决当前管线中的最大痛点（改写器不稳定导致的两条 query 向量路全废），效果可量化
2. Small-to-Big 可之后独立加——不改检索管线，和 HyDE 完全正交
3. 两者可以共存：HyDE 提升检索，Small-to-Big 提升生成，最终效果叠加

## 六、三路召回的完整架构

```
question ──→ ┬─ HyDE: LLM生成假设答案 → embed → 向量检索 top-k₁
              │  (LLM失败→降级跳过)
              │
              ├─ 向量检索: embed原始query → 直接检索 top-k₂
              │
              └─ BM25: jieba分词 → 关键词检索 top-k₃
                                       ↓
                              RRF三路融合(alpha/beta/gamma)
                                       ↓
                                 候选列表 top-K
                                       ↓
                          ┌─ Small-to-Big: 每个chunk换为父文档 ─┐
                          │                                      │
                          ↓                                      ↓
                     rerank(可选)                          喂给LLM生成
```

### RRF 三路权重设计

双路时 RRF 公式：`score = α/(k₁+rank_vec) + β/(k₂+rank_bm25)`

三路时公式：`score = α/(k₁+rank_hyde) + β/(k₂+rank_vec) + γ/(k₃+rank_bm25)`

权重设计的思路：

| 路 | 角色 | 权重倾向 |
|----|------|---------|
| HyDE | 语义对齐的主力 | α 最大（假设答案质量好时） |
| 向量 | 兜底（HyDE 失败时仍可用） | β 中等，与 HyDE 共享密集检索预算 |
| BM25 | 关键词兜底 | γ 保持小值，专门处理专有名词和缩写 |

`alpha`、`beta`、`gamma` 的实际值需要通过 RRF 参数扫描确定（跟双路时一样的扫描方法）。如果 HyDE 路生成质量不稳定（某些 query 的假设答案偏离太大），可以动态调整：HyDE 路与直接向量路共享同一组权重（α+β 固定，内部竞争），避免 HyDE 幻觉对融合结果造成过大影响。

## 七、多路召回的收益递减规律

```
MRR
0.90 │                                     ██ ← 三路
0.85 │                     ████ ████ ████ ██
0.80 │           ████████
0.75 │ ██████████
0.70 │█
     └─────┬─────┬─────┬─────┬─────┬─────
          1路    2路    3路    4路    5路
         (向量) (+BM25) (+HyDE)
```

| 路数 | 组成 | 增量 MRR | 复杂度 | 回报率 |
|------|------|---------|--------|--------|
| 1 路 | 纯向量 | baseline | 低 | — |
| 2 路 | + BM25 | +10-15% | 低 | 极高 |
| 3 路 | + HyDE | +5-10% | 中 | 高 |
| 4 路 | + ColBERT/SPLADE | +1-3% | 高 | 低 |
| 5 路+ | 全上 | 噪声反噬 | 极高 | 负 |

**2-3 路是业界共识的甜点区间。** 超过三路后，每多一路需要多调一个 RRF 权重参数，调参空间指数级增长，且额外的路带来的噪声开始淹没有效信号。

## 八、常见误区

1. **多路召回就是路越多越好**：不是。每多一路引入一个权重参数和新的噪声源。RRF 三路以上权重扫参的时间成本呈指数增长，且不同路的输出高度相关时会出现排名循环（A 比 B 好、B 比 C 好、C 比 A 好）。

2. **HyDE 是查询改写的替代品**：不完全是。HyDE 生成的是假设答案（文档风格），改写器产出的是改写后的 query（问题风格）。两者可以共存——改写器做术语归一化（OA→优化顾问），HyDE 做风格对齐。但在改写器不稳定的情况下，HyDE 单独用比改写器+向量更鲁棒。

3. **Small-to-Big 能提升 MRR**：不能。Small-to-Big 不改检索步骤，只改返回内容。评估 Small-to-Big 的正确指标是 LLM 生成质量（人工评估或自动指标如 ROUGE/BERTScore），不是 MRR。

4. **RRF 权重 k 值必须是 60**：不是。k=60 是原始论文在 MS MARCO 上的经验值。在你的场景下 k 的最佳值可能不同——但 k 值对排名的影响远小于 alpha/beta 权重分配，实际中通常固定 k=60 只调权重。

5. **HyDE 一定会比原始 query 检索好**：不一定。如果 query 本身已经和文档风格高度一致（如 query 就是文档中的原句），直接向量检索可能更好，HyDE 生成的假设答案反而可能引入偏差。所以 HyDE 路和直接向量路要同时保留——HyDE 路作为"风格对齐"的主力，直接向量路作为兜底。

## 九、面试追问预案

### Q1："为什么是三路不是四路？为什么不加 ColBERT？"

**答辩**：收益递减规律——两路到三路的增量约 5-10% MRR，三路到四路可能只有 1-3%。同时三路以上 RRF 的权重参数从 2 个变 3 个，调参成本从 O(n²) 变成 O(n³)。ColBERT 更致命的问题是需要 token 级向量存储——32 个 chunk 变成 ~12,800 个 token 向量，存储和检索成本膨胀 400 倍。在当前数据规模下，这不是功能增强而是架构破坏。

### Q2："HyDE 的 LLM 调用增加了延迟，用户体验怎么保证？"

**答辩**：三条路并行执行——HyDE（LLM 生成 ~300ms + embed ~60ms）、直接向量检索（~60ms）、BM25（~5ms），取最慢的那条决定总延迟。实际增加的是 LLM 生成耗时（200-500ms），但这条延迟在与 BM25 和直接向量并行的情况下，用户体感延迟只增加了 100-200ms（因为它不是串行的）。如果对延迟极度敏感，可以给 HyDE 路设超时（如 500ms），超时自动降级为双路。

### Q3："Small-to-Big 如果父文档很长（2000+ token），超出 LLM 窗口怎么办？"

**答辩**：父文档检索不是返回整个文档，是返回同一个 `##` 标题下的所有子 chunk 拼接。如果你的 `##` 段落平均 token 数就是 300-500 token（之前 chunk_size 选型时已统计），拼接后就是 300-500 token，完全在 LLM 窗口内。如果极端情况下某个 `##` 下有 2000+ token（之前已标记极端长节），加一个硬截断：父文档超过 800 token 时只取前 800 token，或者只取检索命中的 chunk + 前后各 1 个 sibling chunk。

### Q4："HyDE 如果生成了幻觉答案，会不会把检索方向带偏？"

**答辩**：有这个风险，但 HyDE 论文的核心发现是——即使假设答案的 fact 是错误的，它的**语言风格和用词**仍然和领域文档高度一致，因此仍然能把相关文档拉近向量空间。换句话说 HyDE 靠的是风格匹配而不是事实匹配。此外三路设计中 HyDE 路从来不是唯一的路——直接向量和 BM25 仍然在同时检索，幻觉答案拉偏了一个方向还有另外两个方向往回拉。

### Q5："你怎么评估 Small-to-Big 的效果？MRR 不变你怎么证明它有用？"

**答辩**：MRR 确实不变（检索逻辑没变）。正确评估方式有两种：(1) 人工评估——同一组 query，分别用小 chunk 和大 chunk（Small-to-Big）给 LLM 生成答案，人工标注回答的完整度、准确性（1-5 分），做 A/B 对比；(2) 自动评估——如果已有标准答案（ground truth answer），用 ROUGE-L 或 BERTScore 计算生成答案和标准答案的相似度。Small-to-Big 的版本应该在"信息完整度"维度上显著优于小 chunk 版本。
