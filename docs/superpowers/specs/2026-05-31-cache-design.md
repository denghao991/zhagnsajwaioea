# 语义缓存设计

## 问题

RAG 系统中存在大量重复或语义相似的提问，每次提问都走完整流程（Embedding → 检索 → LLM 推理），导致：
- LLM API 调用成本重复消耗
- 相同问题的响应时间无优化
- 用户高频提问场景下累积成本可观

## 解决方案

在 Embedding 和 LLM 调用之间插入语义缓存层，命中时直接返回缓存的 LLM 回答。

### 存储

使用项目已有的 ChromaDB，新增 `cache` collection：

```
collection "documents"  → 文档向量（已有）
collection "cache"      → 缓存条目（新增）
```

每条缓存记录包含：
- `vector` — 问题的 Embedding 向量（用于语义匹配）
- `question` — 原始问题文本
- `answer` — LLM 回答
- `sources` — 召回的来源文档列表
- `refresh_count` — 被用户重新生成的次数
- `timestamp` — 创建时间

复用 ChromaDB 的 HNSW 索引实现 O(log n) 近似搜索，无需自己算余弦相似度。

### 匹配流程

```
用户提问
  ↓
问题向量化（使用已有的 Embedding API）
  ↓
在 cache collection 中 .query(n_results=1)
  ↓
  最接近的距离 < 0.07（≈ 余弦相似度 > 0.93）→ 命中
    检查 refresh_count ≥ 3 → 标记 poisoned，跳过缓存
    返回缓存的 answer，模拟 SSE 推送
  ↓
  未命中 → 走完整检索 + LLM → 存入缓存
```

### 前端适配

- 缓存命中的回答在界面上标记"此回答来自缓存"
- 提供"重新生成"按钮，用户不满意时强制走 LLM
- 请求带 `force_refresh=true` 字段跳过缓存

### 缓存更新

- 用户点"重新生成" → 新回答覆盖该条目的 `answer` 字段，`refresh_count++`
- `refresh_count >= 3` → 标记 `poisoned`，此后语义命中也跳过
- 上传新文档时清空全部缓存（文档变更导致旧缓存失效）

### SSE 兼容

缓存命中时仍使用相同的 `event: token` / `event: context` / `event: done` 推送格式，前端无需感知是否为缓存。`event: done` 中追加 `cached: true` 字段供前端展示标记。

## 未采用的优化方案

以下方案在讨论中被提出但放弃，列出供后续参考：

| 优化 | 放弃原因                                                      |
|------|-----------------------------------------------------------|
| **实体指纹校验** | 从问题中提取关键实体（数字/专有名词）和缓存条目对比，减少语义歧义误命中。实现复杂，新增规则维护成本，且边缘场景极少 |
| **多条目语义簇** | 语义相似的缓存不做覆盖而保留多条。增加存储和匹配复杂度，收益低                           |
| **精确匹配缓存** | 只缓存完全相同的字符串。实现简单但"大量相同的问题"未必都是完全相同的措辞，语义匹配收益更高            |
| **减少 chunk_size / n_results** | 线性降低 LLM 输入 token，降低成本。但与缓存正交，可独立决策                       |
| **引入 Rerank 后减少 chunk 量** | 需引入新模型，与缓存无直接关系                                           |
| **语义分块** | 按段落/标题切分而非固定 token 窗口。与缓存无直接关系                            |
