# Embedding 模型选型方法论

## 一、为什么 chunk_size 之后要选 embedding

chunk_size 决定了检索的最小粒度，embedding 决定了这个粒度上的**语义表达质量**。两者存在强依赖关系：

- 同一个 chunk_size 下，不同 embedding 模型的 MRR 可以差 10-20%
- embedding 的**最大输入长度**反过来约束 chunk_size 的上限——超出模型窗口的块会被硬截断
- embedding 的**向量维度**直接决定 ChromaDB 的存储成本和检索延迟，也影响 API 调用耗时

所以正确顺序是：chunk_size → embedding → RRF 权重。embedding 选了之后，向量路的质量变了，RRF 的 alpha/beta 必须回测。

## 二、评估设计

### 评估管线

```
QA pairs（标注好的问题+正确答案chunk_id）
        ↓
  问题 → 查询改写（LLM）→ embed 问题 → 余弦相似度检索 → 计算 MRR/Recall
        ↓
  文档 → MarkdownChunker 分块 → embed 所有块 → 构建向量矩阵
```

关键设计决策：
- **只走向量检索**，不混 BM25、不混 rerank——纯粹评估 embedding 的独立贡献。混入其他组件后，它们的增益会盖住 embedding 的质量差异
- **查询改写共用同一个 LLM**，但改写结果有非确定性（LLM 温度 > 0），同一 query 跨不同轮次可能产出不同改写，需在分析时标记改写异常
- **batch_size=10**，统计每批的 P50/P95/P99 延迟和总 token 消耗

### 评估配置

| 配置项 | 值 |
|--------|-----|
| 文档集 | 13 篇中文技术文档（OA 产品介绍、故障排查等） |
| Chunk 数 | 32 个（通过 MarkdownChunker 分块） |
| Chunk 总 token | 8,457 |
| Query 数 | 14 条（4 条基础 + 10 条扩充，覆盖 BM25 精准匹配和语义匹配场景） |
| 查询类型 | 用户真实提问风格（口语化、带拼写错误如 "h c s o"、包含冗余寒暄） |

### 指标

| 指标 | 测什么 |
|------|--------|
| **MRR** | 第一个正确答案排在第几位，对排名敏感 |
| **Recall@1/3/5/10/20** | 各 K 值下的命中率 |
| **P50/P95 延迟** | 单 batch（10 条）的 API 调用延迟 |
| **单请求延迟** | per_req_p50 = P50 / batch_size，单块 embedding 耗时 |
| **成本** | API 调用费用（美元），按 input token 计费 |

## 三、评估结果（真实数据）

对 5 个模型配置进行纯向量检索评估：

| 模型 | 维度 | 最大输入 | R@1 | R@3 | R@5 | R@10 | R@20 | MRR | P50(batch) | 单请求 |
|------|------|---------|-----|-----|-----|------|------|-----|-----------|--------|
| text-embedding-v2 | 1536 | 2048 | 0.571 | 0.714 | 0.714 | 0.857 | 0.857 | **0.662** | 0.56s | 0.056s |
| text-embedding-v1 | 1536 | 2048 | 0.786 | 0.929 | 0.929 | **1.0** | **1.0** | 0.854 | 0.47s | 0.047s |
| text-embedding-v3 (512d) | 512 | 8192 | 0.786 | 0.929 | 0.929 | 0.929 | 1.0 | 0.863 | 0.60s | 0.060s |
| text-embedding-v3 (1024d) | 1024 | 8192 | 0.786 | 0.929 | 0.929 | 0.929 | 1.0 | **0.864** | 0.75s | 0.075s |
| text-embedding-v3 (1536d) | 1536 | — | — | — | — | — | — | — | — | — |

注：text-embedding-v3 (1536d) 调用失败——DashScope API 不支持 1536 维输出，报错后自动跳过。

### 加权评分

评估脚本使用复合评分公式：`score = R@5 + MRR - 0.02 × P50`

| 排名 | 模型 | Score | 解读 |
|------|------|-------|------|
| **1** | text-embedding-v3 (512d) | 1.779 | R@5 和 MRR 持平 1024d，但延迟更低 |
| 2 | text-embedding-v3 (1024d) | 1.777 | 几乎并列第一，P50 高 0.15s 拖了后腿 |
| 3 | text-embedding-v1 | 1.774 | 意外强劲，老模型但在 14 条 query 上接近 v3 |
| 4 | text-embedding-v2 | 1.365 | 当前生产 baseline，明显落后 |

**结论：text-embedding-v3 (512d) 为推荐选择**——在质量上与 1024d **无显著差异**（MRR 差 0.001，在同一噪声水平内），但以一半的维度带来更低的延迟（P50 0.60s vs 0.75s）和更少的存储。这不是"512d 质量优于 1024d"的结论（统计上不成立），而是"质量平手时选更便宜的方案"的工程决策。

**评分公式的敏感性分析**：`score = R@5 + MRR - λ × P50`，λ 取不同值时排名变化：

| λ | v3-512d | v3-1024d | v1 | v2 | 结论稳定性 |
|---|---------|----------|-----|-----|-----------|
| 0（忽略延迟） | 1.792 | 1.793 | 1.783 | 1.376 | v3-1024d 微弱反超 0.001 |
| 0.02（当前） | 1.779 | 1.777 | 1.774 | 1.365 | v3-512d 推荐 |
| 0.05 | 1.762 | 1.756 | 1.760 | 1.348 | v3-512d 领先扩大 |
| 0.10 | 1.732 | 1.717 | 1.736 | 1.320 | **v1 反超**（延迟最低） |

无论 λ 如何变化，四个模型的**相对集团不变**：v1 + 两个 v3 构成第一集团，v2 单独落后。因此最终结论对权重不敏感——选哪个 λ 都轮不到 v2。面试时被追问权重依据，可以用这个敏感性分析证明"结论稳健"比"权重精确"更重要。

### 维度对 v3 质量的影响

| v3 维度 | MRR | R@5 | P50 | 存储（32 块） |
|---------|-----|-----|-----|-------------|
| 512d | 0.8626 | 0.929 | 0.60s | ~10 KB |
| 1024d | 0.8636 | 0.929 | 0.75s | ~20 KB |

**维度翻倍，MRR 几乎不变（Δ = 0.001）**。这说明在 32 块的小规模下，512 维已经足够编码当前文档集的语义信息。1024 维的额外容量在当前数据上无法转化为检索质量提升。

但需注意：这个结论受限于 32 块的数据规模。如果文档量和语义多样性大幅增加（如扩展到数万块），1024 维的优势可能会体现出来。面试时可以主动说明这是一个"当前数据下的结论，规模化后需要回测"。

## 四、关键陷阱：查询改写的非确定性

### 现象

v2 的 MRR 只有 0.662，比 v1 低 0.192。但深入 per-question 分析发现，**14 条 query 中有 2 条（14%）被查询改写器彻底破坏**：

| Query | v2 改写结果 | 改写质量 | v2 Rank |
|-------|-----------|---------|---------|
| "OA监控消息，我手机收不到告警短信，是不是咱们OA后台把我的手机号给拦截了？帮忙看一下。" | **"优化"** | 🔴 灾难，丢失全部语义 | null（未找到） |
| "OA上监控配置完成之后，是多长时间会触发监控告警？我们这边配置了一个一定会触发的告警规则，可是等了一会，也没有相应的告警。" | **"优化顾问(OA)"** | 🔴 同样灾难 | null（未找到） |

而同样两条 query 在 v1 和 v3 的评估轮次中，改写器保留了完整的原始问题文本，对应的正确答案都排到了第 1 位。

### 修复分析

排除这两条改写异常 query 后：

| 模型 | 14 条 MRR | 12 条 MRR（排除改写异常） | 差距 |
|------|----------|------------------------|------|
| text-embedding-v2 | 0.662 | **0.772** | +0.110 |
| text-embedding-v3 (512d) | 0.863 | 0.863 | 0 |
| text-embedding-v3 (1024d) | 0.864 | 0.864 | 0 |

v2 的"真实"MRR 约 0.77，与 v3 的差距从 0.202 缩小到 0.09。改写器引入的噪声让 v2 的评估结果显著偏低了。

### 方法论教训

1. **per-question 分析是必须的**——只看汇总 MRR 会掩盖改写器的不稳定行为
2. **改写器的非确定性可能成为 embedding 评估的混杂变量**——同一 query 跨不同模型轮次的改写结果不同，因为 LLM 调用有随机性
3. **修复方案**：先批量改写所有 query 并固化（缓存改写结果），再用同一批改写后 query 评估所有 embedding 模型。或者设置 temperature=0 消除非确定性

## 五、方法论：三层漏斗

```
市面上 ~20 个中文 Embedding 模型
          ↓
    第一层：最大输入长度（硬约束）
          ↓
    第二层：维度与成本（预算约束）
          ↓
    第三层：MRR 评估（质量决策）
          ↓
   最终 2-3 个候选 → 加权评分 → 选最优
```

### 第一层：硬约束——最大输入长度

模型窗口必须 ≥ chunk_size × 1.5。以 chunk_size=384 为例（本项目实际值）：

| 模型 | 最大输入长度 | 384 token 场景下可用？ |
|------|------------|---------------------|
| text-embedding-v1/v2 | 2048 | ✅ 5× 余量 |
| text-embedding-v3 | 8192 | ✅ 21× 余量 |
| BGE-large-zh-v1.5 | 512 | ⚠️ 1.3× 余量，刚好擦边 |
| m3e-base | 512 | ⚠️ 同上 |

当 chunk_size 较大（如 512-1024）时，BGE-large-zh 和 m3e-base 的直接淘汰就变得有依据——它们是 512 窗口模型，在 P75 接近上限的场景下不可靠。

### 第二层：预算约束

| 模型 | 维度 | 单价 ($/M token) | 32 块成本 |
|------|------|-----------------|----------|
| text-embedding-v1 | 1536 | 0.0007 | $0.000006 |
| text-embedding-v2 | 1536 | 0.0007 | $0.000006 |
| text-embedding-v3 | 512-1024 | 0.0005 | $0.000004 |

在本项目规模下（32 块），成本的绝对值差异微乎其微（$0.000002），不作为淘汰依据。但在百万级块规模下，成本差异会放大到可感知的程度（$0.02 vs $0.03）。

### 第三层：质量评估

纯向量检索 + MRR 复合评分。本项目结论：**text-embedding-v3 (512d) 胜出**。

## 六、常见误区

1. **维度越高越好**：v3 (512d) 和 (1024d) 的 MRR 差异仅 0.001，在该数据集上几乎无差异。维度是容量上界，不是质量保证。

2. **用 MTEB 榜单排名直接选模型**：MTEB 是通用评测，不代表你的特定文档和查询分布。必须在自己的数据上跑 MRR。

3. **v2 > v1 一定成立**：不。v1 在 14 条 query 上的 MRR（0.854）比 v2（0.662）高 0.192。这可能是因为 v2 的训练数据或对比学习策略在中文客服类 query 上反而退化了。版本号大不代表在你的场景上更强。

4. **换 embedding 不需要重新调 RRF 权重**：错。换了 embedding 后向量路的质量变了——v3 的向量检索 MRR 比 v2 高 0.2，这意味着向量路更强了，RRF 中应该给向量路更大权重（提高 alpha）。

5. **汇总 MRR 就够了，不需要 per-query 分析**：v2 的 MRR 被两条改写异常的 query 拉低了 0.11。没有 per-query 分析就会被误导。

## 七、方法论的边界与局限

面试官如果追问"你这方法有什么问题"，不要防御，主动亮出以下五个已知局限和对应的改进路线。主动承认局限比强辩更能获得面试官尊重。

### 1. 缺乏绝对质量基线

**当前缺陷**：评分公式只做相对排序，没有回答"1.78 分算好还是差"。缺少三个锚点：
- **下限**：BM25-only 的 MRR（最差能差到哪）
- **上限**：人工标注的理想排序 MRR（理论上限在哪）
- **业务阈值**：使用者可接受的最低 R@5 / MRR（SLA 基准线）

**改进路线**：跑一次 BM25-only 作为 trivial baseline（成本极低，只需在评估脚本里跳过向量检索），人工标注理想排序（需额外 1-2 小时），从产品需求文档反推业务 SLA。

### 2. 改写器与 embedding 的耦合未解耦

**当前缺陷**：评估的是"改写器 + embedding"的联合效果。如果改写器对某个 embedding 的语义空间有系统性偏好（比如总是产出对 v1 更友好的短 query），模型排名里有一部分是改写器的偏差。没有做消融实验（固定改写结果 × 固定 embedding）来分解方差来源。

**缓解因素**（非辩护，是弱缓解）：四个模型共享同一套改写结果，改写器的偏差对四个模型施加的是同向压力——它可能压低绝对 MRR，但不太可能逆转相对排名，除非某个模型对特定改写模式极其敏感（v2 就是这种情况——而这本身就是重要发现）。

**改进路线**：先批量改写全部 query 并缓存，消除跨轮次不确定性；再用 temperature=0 跑一版作为对照，量化改写非确定性的影响幅度。

### 3. 纯向量评估 vs 混合检索的张力

**当前缺陷**：生产中是 hybrid 搜索（BM25 + 向量 + RRF），但评估只用纯向量检索。一个在纯向量下表现好的 embedding，在 hybrid 场景下可能和 BM25 路的互补性较差——BM25 擅长关键词匹配，如果 embedding 也侧重关键词级语义，两者信息重叠高，RRF 合并后的增益有限；若 embedding 侧重段落级语义，与 BM25 的互补性更强。纯向量评估无法检测这个效应。

**当前选择的理由**：纯向量评估是 necessary condition，不是 sufficient condition——先筛掉明显差的（v2），再用最优候选回测 RRF 混合检索。全因子设计（embedding × RRF 权重 × BM25 配置）会导致实验矩阵爆炸，在 32 个 chunk 的场景下过拟合风险很高。

**改进路线**：embedding 选定后，额外跑一次 hybrid 下的 MRR 作为确认性验证（而非选型依据），成本是额外一次 RRF 参数扫描。

### 4. 评估范围仅限于单一供应商

**当前缺陷**：只测了阿里云 DashScope 的三个模型。BGE-M3、Cohere Embed v3、Voyage-2 等没有进入评估。这不是因为它们不好，而是因为当前阶段的目标是**供应商内选型**（已有 DashScope 采购关系，选最优模型），不是**跨供应商选型**（从零开始对比所有服务）。

**改进路线**：明确区分两个选型阶段——
- Phase 1（已完成）：供应商内选型，在 DashScope 内确定 v3-512d
- Phase 2（未做）：跨供应商对比，候选池加入 BGE-M3（自部署或 ModelScope API）、Cohere、Voyage，用同一套 query 和 chunk 跑相同评估管线

面试官追问"为什么没测 BGE-M3"时，直接说明 Phase 1/Phase 2 的区分，而不是回避问题。

### 5. 评估管线假设单向流水线，实践需要迭代

**当前缺陷**：方法论流程画成 chunk_size → embedding → RRF 三步单向流水线。但实际上 upstream 选择完成后，如果下游（RRF）的表现偏离预期，可能需要回到 upstream 重新决策。

**改进路线**：在实践中，如果 embedding 选完后 RRF 的最优权重发生了剧烈变化（比如最优 alpha 接近 0，说明向量路几乎没贡献），那应该回溯检查 embedding 是否有问题。这个反馈回路在面试中直接提出来就是加分项——说明你理解评估是一个有反馈的决策过程，不是一次性的单向流水线。

## 八、面试追问预案

### Q1："为什么 512 维和 1024 维几乎一样？什么时候 1024 维才有价值？"

**答辩**：32 块、8,457 token 的数据量下，512 维已经足够覆盖语义多样性。当数据规模增长到数万块、语义类型急剧增多时，512 维的 embedding 空间会变得拥挤（不同语义的向量被压缩到过于相近的位置），此时 1024 维的优势才会体现。这是一个"数据规模驱动的维度需求"问题，不是 512 < 1024 就永远成立。

### Q2："v2 为什么比 v1 还差？版本升级不是应该更好吗？"

**答辩**：v2 在通用评测上确实比 v1 好，但在本项目的中文客服类 query 场景下反而退步了。可能原因：(1) v2 的对比学习策略优化了其他类型 query 但对口语化、带噪声的中文 query 不够鲁棒；(2) 两次改写失败让 v2 的评估结果偏低（排除后 MRR 0.77 vs 0.85）。这说明不能盲信版本号，必须在自己数据上实测。

### Q3："你的评估只有 14 条 query，结论可靠吗？"

**答辩**：14 条确实是小样本，统计效力有限。但(1) v3 相对 v2 的 MRR 差异达 0.2，即使排除改写异常后仍有 0.09，在实际工程中是显著的量级；(2) 14 条 query 覆盖了 4 种类型，有基本类型覆盖；(3) "512d 和 1024d 质量无差异"的结论可以补做 Bootstrap 验证（1000 次重采样看 CI 是否跨越零），如果 CI 包含零则用数据支撑"无差异"而非靠嘴说；(4) 更根本的改进是增加 query 到 30-40 条并锚定绝对基线（见第七节）。坦承样本量限制 + 说出具体多少条够 + 说明统计检验方法，三者缺一不可。

### Q4："查询改写引入了噪声，你怎么修正？"

**答辩**：两个方案——(1) 快捷方案：在所有模型评估前，先批量改写全部 query 并固化结果（缓存改写），消除跨轮次的非确定性；(2) 根本方案：设置 embedding 评估专用 LLM 的 temperature=0，或者使用确定性的改写策略（规则 + 词典）。这些改动都可以在 10 分钟内完成。

### Q5："你为什么不测 BGE-M3？业界公认很强"

**答辩**：区分两个选型阶段——Phase 1（已完成）是供应商内选型，我已有 DashScope 采购关系，先在已有供应商内确定最优模型（即 v3-512d）；Phase 2（未做）是跨供应商对比，候选池加入 BGE-M3（自部署或 ModelScope API）、Cohere、Voyage 等。BGE-M3 的优势在跨语言和多粒度，在 384 token 纯中文场景下不一定优于 v3-512d，但这个结论需要用同一套评估管线跑出来才能说——我已将其列入 Phase 2 的候选池。评估优先级的逻辑是"先优化已有资源，再探索新资源"，不是"BGE-M3 不好所以没测"。

### Q6："你的评分公式权重为什么是 0.02？换一个值结论会变吗？"

**答辩**：0.02 等价于"1 秒额外延迟 = 0.02 分"，来自工程经验（对用户体验损失的量化）。但我做了敏感性分析——λ 从 0 变到 0.10，四个模型的相对集团不变（v1 + v3 始终是第一集团，v2 始终垫底），所以具体权重不影响最终决策方向。关键是"结论对参数不敏感"这个论证，而不是"参数精确"。"如果你要求更严谨的权重，可以从业务指标反推——比如用 P95 延迟每增加 100ms 导致的用户流失率来校准 λ。"
